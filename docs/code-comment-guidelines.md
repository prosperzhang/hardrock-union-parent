# 中文注释规范

## 目标

这份规范用于约束 `hardrock-union` 项目的中文注释写法，目标是提升代码可读性，同时避免“注释比代码还多、而且很快过期”的维护问题。

## 基本原则

1. 注释解释“为什么”和“业务规则”，不要逐行翻译代码。
2. 注释优先写在类、方法和复杂分支上，不要求每一行都写。
3. 注释要和当前业务模型一致，避免保留历史说法。
4. 能通过更清晰的方法名、字段名表达清楚的内容，优先改命名，不靠注释补救。

## 推荐写法

### 类注释

适用场景：
- Service
- Facade
- Controller
- 复杂 DTO / Entity / Enum

建议说明：
- 这个类负责什么
- 它服务哪个 app / 业务域
- 它和其它服务的边界是什么

### 方法注释

适用场景：
- 对外暴露的核心业务方法
- 包含事务、状态流转、跨模块协作的方法

建议说明：
- 输入输出是什么
- 关键业务校验是什么
- 方法执行后会产生什么副作用

### 关键分支注释

适用场景：
- 状态机流转
- 库存回滚
- 幂等处理
- 租户隔离
- 跨 app 对象映射

建议说明：
- 这里为什么这样判断
- 不这样做会出现什么业务问题

## 不推荐写法

- 不要给每一行赋值都写注释。
- 不要写“把值设置给字段”这类重复代码表面的注释。
- 不要把已经过时的架构认知继续写进注释里。
- 不要用大段注释代替拆方法。

## 当前项目落地建议

- `platform-iam`：
  重点解释 app 级登录、租户隔离、角色校验。
- `business-merchant`：
  重点解释报价单、订单、库存扣减、报价转订单。
- `business-project`：
  重点解释项目、工地、以及与 `primeload-marketplace` 交易目标的关系。
- `solution-*`：
  重点解释“应用装配层”职责，不在这里堆通用业务规则。
